# 9. Incremental Load Logic

For each tenant, call all 3 SPs with @ReconModeInd = 0 and @LastRunDate from YOYFactPipeline. Group all SP results by EnquiryNo.

  For each EnquiryNo in the extract:

    1. Compute DataHash:
       - Include ALL SP columns in the hash (including columns excluded from the fact table). The hash detects any upstream change — it is not limited to stored columns.
       - SP1 values (single row, all columns, pipe-delimited)
       - SP2 rows sorted ascending by CountryCode (all columns, pipe-delimited)
       - SP3 rows sorted ascending by TagName (all columns, pipe-delimited)
       - Concatenate all three → SHA-256 hash
       - Sort order is mandatory for hash determinism
       - Represent NULL values as the literal string "NULL" in the pipe-delimited concatenation to ensure hash determinism.

      Pre-filter: if SP1's BookingValue = 0 AND NoOfPax = 0 (the total enquiry-level values):
       → If EnquiryNo NOT in tracking table → skip entirely
       → If EnquiryNo IS in tracking table → go to Case E
    
    2. Look up TenantId + EnquiryNo in dhruvlog.dbo.EnquiryParquetFile → Cases A–E

    Case A — Not in tracking table (new booking):
      → Write to fact_bookingcountries and fact_bookingtags partitions
      → If SP3 returns no rows for an EnquiryNo, write nothing to fact_bookingtags for that EnquiryNo. This is valid — not all bookings have tags.
      → INSERT into EnquiryParquetFile after successful write

    Case B — Found, DataHash matches:
      → No change. Skip entirely.

    Case C — Found, DataHash differs, ParquetFile matches current BookingMonth:
      → Remove all rows for EnquiryNo from current partition in all fact files
      → Write updated rows
      → UPDATE EnquiryParquetFile (DataHash, UpdatedOn)
      → If SP3 returns no rows for this EnquiryNo, write nothing to fact_bookingtags (same rule as Case A applies throughout).

    Case D — Found, DataHash differs, ParquetFile does NOT match current BookingMonth (BookingDate has changed):
      → Use the stored ParquetFile path from EnquiryParquetFile to locate the old partition
      → Read the old partition, remove rows for this EnquiryNo (if any), then rewrite old partition
      → Write updated rows to new BookingMonth partition
      → UPDATE EnquiryParquetFile (ParquetFile, DataHash, UpdatedOn)
      → If SP3 returns no rows for this EnquiryNo, write nothing to fact_bookingtags (same rule as Case A applies throughout).
      → The stored path references fact_bookingcountries. The equivalent fact_bookingtags path uses the same BookingMonth partition — derive it by substituting fact_bookingtags for fact_bookingcountries in the stored path.

     Case E — Found, BookingValue = 0 AND NoOfPax = 0 (hard delete):
      → Use stored ParquetFile path to locate partition, remove rows from both fact files
      → DELETE EnquiryNo from EnquiryParquetFile
      → The stored path references fact_bookingcountries. The equivalent fact_bookingtags path uses the same BookingMonth partition — derive it by substituting fact_bookingtags for fact_bookingcountries in the stored path.

  Case C/D sync gap — if the tracking table says a row exists but the Parquet file doesn't contain it (e.g., previous partial failure), the remove step silently does nothing and the insert proceeds correctly. This is actually fine and self-healing.

  Consistency rule: always write to EnquiryParquetFile AFTER the Parquet
  write succeeds. On failure, log the EnquiryNo — it will re-process on
  the next run as hash will differ.

  Write to temp file first, then atomically rename on success (applies to
  all partition writes above).

  The SPs are designed so that if any data for an EnquiryNo changes (header,
  country, or tag), all 3 SPs return that EnquiryNo. This is a SP design contract — the pipeline relies on it.

# 10. Snapshot Logic

## Daily snapshot

Filter:

* BookingDate ≤ SnapshotDate
* DepartureDate ≥ SnapshotDate

---

## Relative Month

* Month 1 = current calendar month
* Month 2 = next month

## Note

Snapshots are built by reading fact_bookingcountries and fact_bookingtags
Parquet files and applying the snapshot filter. Each run creates a new SnapshotDate=YYYY-MM-DD partition — this is always a full rebuild for that date, not incremental. No hash tracking is needed for snapshot files.

Retain the last N days of snapshot partitions (configurable via snapshot_retention_days in config.yaml, default 30). Delete older SnapshotDate = partitions after a successful run.

# 11. Snapshot Tables

## A. Country snapshot

```
{DATA_ROOT}/tenant_id={tenant_id}/fact_bookingcountries_snapshot/SnapshotDate=YYYY-MM-DD/
```

Group by:

* SnapshotDate
* BookingFY
* DepartureFY
* RelativeDepartureMonth
* TourType
* TourGenericCode
* CountryCode
* CountryName

Measures (aggregate):

* SUM(BookingValue) AS BookingValue
* SUM(NoOfPax) AS NoOfPax

### Note

RelativeDepartureMonth is computed at snapshot build time as the number of calendar months from SnapshotDate's month to the row's DepartureMonth, plus 1. Month 1 = current calendar month (the month containing SnapshotDate). This is not stored in the fact table.

---

## B. Country + Tag snapshot

```
{DATA_ROOT}/tenant_id={tenant_id}/fact_bookingcountries_snapshot_country_tag/SnapshotDate=YYYY-MM-DD/
```

Group by:

* SnapshotDate
* BookingFY
* DepartureFY
* RelativeDepartureMonth
* TourType
* TourGenericCode
* CountryCode
* CountryName
* TagName

Measures (aggregate):

* SUM(BookingValue) AS BookingValue
* SUM(NoOfPax) AS NoOfPax

### Note

Country + Tag snapshot is built by joining fact_bookingcountries with fact_bookingtags on EnquiryNo, applying the snapshot filter to both, then grouping as defined. Use an INNER JOIN — only EnquiryNos present in both tables are included in Snapshot B. Bookings with no tags do not appear in this snapshot.

Use fact_bookingcountries.BookingValue and fact_bookingcountries.NoOfPax as the measure values. fact_bookingtags.BookingValue is the full enquiry-level value and must not be used in Snapshot B aggregation.

RelativeDepartureMonth is computed at snapshot build time as the number of calendar months from SnapshotDate's month to the row's DepartureMonth, plus 1. Month 1 = current calendar month (the month containing SnapshotDate). This is not stored in the fact table.

# 20. Amends to Main enquiry record
  Enquiry's all values can change. Booking value changes are expected and is normal.
  Other values rarely change, but can happen:
  ClientCode, TourType, TourGenericCode, DepartureDate, ReturnDate, EnquiryDate, BookingDate, NoOfPax.
  Also, CountryCodes and TagNames.

  The EnquiryParquetFile tracking table (Section 21) handles real-time data
  integrity during every daily pipeline run, covering:
  - CountryCode or TagName added/removed (hash changes → full EnquiryNo re-process)
  - BookingDate amended (ParquetFile path mismatch → partition move, no SP call needed)
  - Duplicate booking cancellations (zero-value → hard delete from Parquet and tracking table)
  - The separate reconciliation job (reconciliation_job.py) using @ReconModeInd = 1 is a weekly safety net only. It is not the primary mechanism for data correctness.

  - reconciliation_job.py goes in the Manual folder. Do not over-engineer — clear and readable is preferred.

# 21. Enquiry Parquet Tracking Table

  Maintain dhruvlog.dbo.EnquiryParquetFile to enable hash-based change
  detection during every pipeline run, ensuring real-time data integrity
  without periodic batch scanning.

  DDL:
  CREATE TABLE dhruvlog.dbo.EnquiryParquetFile (
    SeqNo       int           NOT NULL identity(1,1),
    TenantId    nvarchar(50)  NOT NULL,
    EnquiryNo   int           NOT NULL,
    ParquetFile nvarchar(500) NOT NULL,  
    DataHash    nvarchar(64)  NOT NULL,  -- SHA-256 hex string
    CreatedOn   datetime2     DEFAULT (getutcdate()) NOT NULL,
    UpdatedOn   datetime2     DEFAULT (getutcdate()) NOT NULL,
    constraint PK_EnquiryParquetFile PRIMARY KEY (SeqNo)
  );
  create unique index IX_EnquiryParquetFile_01 on dhruvlog.dbo.EnquiryParquetFile(TenantId, EnquiryNo);

  Rules:
  - ParquetFile column stores the path relative to {DATA_ROOT}/tenant_id={tenant_id}/, e.g. fact_bookingcountries/BookingMonth=2025-11
  - One row per TenantId + EnquiryNo (each enquiry has exactly one BookingDate
    → one BookingMonth → one partition)
  - ParquetFile stores relative partition path — used to detect partition moves
  - DataHash covers all SP values for the EnquiryNo across all 3 SPs
  - Always include TenantId in all queries to maintain tenant isolation
  - Use the same dhruvlog connection as LastRunTracking (same server, same
    decrypted connection string)
